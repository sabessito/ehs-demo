# 요구사항 체크리스트 — 스마트 안전기술 기획 면접용 앱

> 하영우님이 대화 전체에서 요청한 사항을 누적 정리. 각 항목의 이행 여부는
> "제작자(Builder)"가 구현하고 "검토자(Reviewer)"가 독립 검증한다.

## ◎ 최상위 핵심 메시지 (모든 것보다 우선 — K 섹션)
> 이 앱의 존재 이유. 인트로와 앱 전체 구성이 이 3꼭지를 전달해야 한다.
- [K0] 앱 목적: "삼성EHS전략연구소에서 일한다면, 데이터로 현황을 정확히 진단하고
       → 전략을 수립하고 → 이슈를 과제화·기획안 작성 → 로드맵을 그려 각 협력
       삼성 관계사에 기획·구축까지 하는 업무 전체"를 샘플로 보여준다.
- [K1] (꼭지1) 데이터 현황분석: 내부+외부 데이터를 모두 활용.
       - 외부 = 중대재해 법규, 타사 사례, 해외 사례 리서치
       - 내부 = 센서·설비·장비·인사정보 등 모을 수 있는 데이터
       - 내·외부 데이터를 '어떤 관점으로 묶고', 그 관점에 따라 '어떻게 데이터화해
         분석'할지 보여준다.
- [K2] (꼭지2) 전략 방향 수립 Framework — **Top-down과 Bottom-up을 동시에 고려**:
       - Top-down: 삼성 그룹의 사업별 분류 기준 + 해당 사업의 현재 전략적 방향성을
         기준으로 전략 Framework 구성
       - Bottom-up: 실제 데이터 현황 진단으로 '예측되는 중대재해 분류'에 의해
         영향받는 관계사들을 묶어 Framework 구성
       - 두 접근의 교차점에서 Theme별 전략·우선순위를 도출.
- [K3] (꼭지3) 과제 기획 → 구축 → 운영:
       - 전략→과제 기획 방법론 관점 제시
       - 로드맵으로 방향을 한눈에 조망하는 Framework/기준 제시
       - KPI 관리(운영 지표 + 재무 지표)까지 제시.
- [K4] 인트로 페이지 신설: 앱 전체 흐름 + 보는 방법 + 위 목적/3꼭지 요약.
- [K5] 앱의 탭 구성도 위 3꼭지 흐름을 따르도록 재검토·정렬한다.

## ◎ 검토 체제 (이번 작업) — 5인
- [REV] 제작자(Builder) 1 + 검토자 4:
  - R1 전략·기획 검토: K2/K3 메시지가 전략기획 관점에서 설득력 있는가
  - R2 데이터Key·View 검토: 데이터 Key 관점이 타당한가, 어떤 View가
       면접관을 'Wow'하게 할지
  - R3 데이터 정합성 검토: 수치·출처·계보·원본보존(G5)이 정확/일관한가
  - R4 용어·문구 검토: 단어 하나하나가 전략적·기획적 언어인지(사소·일상어가
       아닌 컨설팅/전략 용어로 표현됐는지) 점검

## ★ 상시 제약 (모든 작업에 항상 적용)
- [G1] **모든 개발 업무는 데스크탑/노트북 없이 '폰'으로만 수행한다.**
       (git, 배포, 코드 수정, 확인까지 전부 모바일 기준으로 안내·설계할 것)
- [G2] **Claude는 작업을 시작하기 전 항상 이 요구사항 파일을 먼저 확인하고,
       반영했는지 점검한 뒤 실행한다.** (제작자/검토자 교차검증 유지)
- [G3] **산출물 전달 규칙**: Claude는 폰 내부 폴더(내 파일>개발>스마트 안전 기획)에

       묶어 제공하고, 변경분(생성/수정/삭제)을 CHANGELOG로 명시한다. 사용자는 이를
       내려받아 해당 폴더에 압축 해제하여 동기화한다.

       (5) 추진 로드맵 (6) Cost·Limit + 데이터 아키텍처(통합) (7) 접속·QR

- [G5] **★★ 원본 데이터는 절대 수정해서 표현하지 않는다 ★★**

       가공은 원본을 건드리지 않고 '정렬/속성추가/통합'의 단계로만 표현한다.
       (Phase 구조는 J 섹션 참조) 이 규칙은 최우선으로 지킨다.
- [G6] **UX 요구(하영우님 추가 기능)**:
       - 전문용어·줄임말·수식은 탭(클릭) 시 뜻/설명이 뜨는 용어풀이(tooltip/popover) 제공
       - 페이지 전환은 이전/다음 버튼이 아니라 좌우 스와이프(스윕)로 넘긴다
       - 헤더의 단계 아이콘 제거. 그 자리에 국내/해외 토글을 '국기만' 작은 아이콘으로
       - 스텝(탭)은 직접 클릭 가능해야 한다(탭 스트립 유지)

## A. 앱 기본 (1차 요청)
- [A1] 모바일(폰)에서 보는 소개용 앱
- [A2] EHS 전략 Full Process를 단계별로 보여줄 것:
       데이터 분석 → 이슈 발굴 → 전략 → 핵심 과제(기획) → 구축/로드맵
- [A3] 인터넷/중대재해 법규 기반의 샘플 데이터 활용
- [A4] 계열사(협력 기업) 전파/컨설팅 관점이 드러날 것

## B. 데이터 분석 3대 관점 (2~3차 요청)
- [B1] 데이터를 3가지 큰 관점(오브젝트)으로 표현
- [B2] 오브젝트 = Cost(예방투자) / Risk Index(위험도 지수) / Limit(운영기준선)
- [B3] 관점: **중대재해 0이 목표**. 사고는 조치가 있었음에도가 아니라 조치·기준
       누락에서 발생 → 무엇을 놓쳤는지 데이터로 찾아 선제 개선. (구 표현 "이윤추구
       적정선 줄타기"는 폐기 — 안전을 비용편익으로 환원하지 않는다)
- [B4] ALARP 프레임워크를 음영(영역)으로 시각화
- [B5] 법적 최소 기준선을 별도 표시(중대재해법은 비용 과다 항변 불인정)

## C. 실제/정부 데이터 (4~5차 요청)
- [C1] 샘플 데이터가 "무엇인지" 명확히(출처/대상 명시)
- [C2] 특정 기업·특정 영역 "전체"를 보여주듯 단일 사업장 케이스화
- [C3] 한국 기준: 고용노동부 2024 재해유형 분포 반영
- [C4] 큰 그림: 전체 노동자 수 / 사망자 수 / 부상(사고재해자) 수 함께 표시
- [C5] 해외 사업장 케이스 별도 확보(비중 항목이 한국과 다름)
- [C6] 해외 케이스도 한국과 "동일 기준"으로 정리
- [C7] 해외 케이스 중 자국법 위반/중대재해법 상당 상황·사고 확인
- [C8] 앱에서 한국/해외 케이스를 "선택 가능"하게 토글 구현

## D. 프로세스 (5차 요청)
- [D1] 요구사항 리스트 파일 작성(이 파일)
- [D2] 제작자/검토자 두 역할로 교차 검증, 더 나은 결과 제시

## H. 데이터 모델 / Key 설계 (데이터 아키텍처 심화)
- [H1] 최소 단위 = 사건(incident) 및 피해자(victim). 피해자가 별도 최소 단위인
       이유는 보상 체계가 '사람 단위'이기 때문.
- [H2] Key 계층: root_cause_id ← incident_id ← victim_id
       (근본원인 1 : 사건 N : 피해자 M)
- [H3] 분석 계층: 현상(사건 발생형태) → 직접원인(cause) → 근본원인(root cause)
- [H4] 근본원인 해결 시 연결된 모든 사건이 해소되는 구조. 따라서 사건별
       근본원인 분석(RCA)을 추가 수행한다.
- [H5] RCA 방법론 = Fishbone(특성요인도) 6M 분류
       (Man/Machine/Material/Method/Measurement/Environment)
- [H6] **Cost·Limit·Risk Index는 사건이 아니라 '근본원인' 단위에 연결**한다.
       - Cost = 해당 근본원인을 제거하는 대책 비용(중복 제거)
       - Risk Index(root_cause) = Σ(연결 사건의 빈도 × 피해 심각도)
       - ALARP 판정도 근본원인 단위로 수행 → 비용 대비 위험감소 큰 순으로 투자
- [H7] 각 원본 데이터셋이 어떤 Key로 조인되고 어떤 가공을 거쳤는지 명시한다.

## I. Cost 정량 산출 & Limit 현실화 (데이터 심화 2)
- [I1] Cost는 근본원인 제거 '대책'을 실제 장비/조치 단위로 분해해 정량 산출한다.
       산식: Σ(단가 × 수량/개소). 예: CCTV 개소, 스마트 헬멧 인원수, 안전난간 m 등.
- [I2] 장비 단가는 시연용 추정 단가표를 명시(실제 견적 시 교체). 출처/가정 표기.
- [I3] Limit의 학술적 근거:
       - HSE 최대 허용 위험: 작업자 연 1/1,000, 일반인 연 1/10,000 (상한선=Unacceptable)
       - ALARP 비용편익: VPF/VSL(통계적 생명가치, 영국 약 £1~2M) 기반 CBA
       - 정밀 정량분석 필수 아님 — 전문가 판단 + 조잡한 CBA로 충분(HSE 정책)
- [I4] 학술 기준을 한국 현실로 즉시 못 옮기는 부분은 '현실화 로드맵'으로 둔다.
       Limit은 산업 특성·사업 전략에 따라 변동성이 크다는 점을 명시.
- [I5] Limit 현실화 전략 케이스 2~3개를 앱에 적용(케이스별 Limit 조정 로직):
       (a) 해외 다발 구축 — 공사안전 포커싱(추락/중장비 가중)
       (b) 노후 설비 운영 — 협착/화재 가중, 예지보전 연계
       (c) 신규 양산 가동 — 미숙련 인력(Man) 가중, 교육·동선 관리
- [I6] 각 케이스는 6M 가중치를 다르게 적용 → 같은 사건도 Limit이 달라짐을 보인다.

## J. 데이터 정리 3단계 (원본 보존 — 하영우님 정의)
> 핵심 원칙: 원본은 절대 수정하지 않는다(G5). 아래는 원본을 '복제·변환'하는 단계.
- [J0] 원본(Raw): KOSHA 마이크로데이터, 사고사례 API, BLS 등 — 그대로 보존, 불변.
- [J1] Phase 1 — 정렬(Sort/Pivot): 원본을 '사건(incident)'과 '사람(victim)' 관점으로
       정렬·재구성. 키 포인트가 사건·사람이므로 이 두 축으로 사건들을 정리.
- [J2] Phase 2 — 속성 추가(Enrich): 근본원인·현상 기준으로 사건에 속성을 추가.
       (현상→직접원인→근본원인, 6M 분류 등 분석 컬럼을 '추가'하는 단계)
- [J3] Phase 3 — 통합(Integration): Phase1·2로 가공된(원본 복제본) 데이터들을
       하나로 통합. 이 통합 결과가 앱의 분석/Cost/Limit 산출 기반이 된다.
- [J4] 각 Phase는 원본을 덮어쓰지 않고 별도 산출물로 남긴다(계보 추적 가능).

### 검토자가 잡은 한계 (정직한 고지)
- Cost·Limit·Event 가중치는 내부 운영 데이터 성격이라 전부 가상값. 면접 시 "구조 시연용"임을 명시할 것.
- 해외 비치명 부상 약 260만건은 추정치(민간 전체). 정확 수치는 BLS SOII 별도 확인 필요.
- 실제 마이크로데이터(KOSHA 13.8만건)로 업종 필터 집계를 원하면 XLSX 업로드 시 반영 가능.
